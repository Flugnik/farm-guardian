# 1. Видение проекта

**Farm Guardian** развивается из Telegram-бота в систему управления фермерским хозяйством.

Цель проекта — создать **единый цифровой центр управления фермой**, который:

- принимает команды и наблюдения от человека
- фиксирует события жизни хозяйства
- хранит память о животных и процессах
- помогает принимать решения
- предупреждает о рисках (например, погодных)

В долгосрочной перспективе система должна стать **"операционной системой фермы"**.

---

# 2. Основные принципы архитектуры

## Local-first

Система должна работать **локально**, без обязательной зависимости от облачных сервисов.

Интернет может использоваться для:

- погоды
- обновлений
- внешних моделей

Но основная логика должна работать без него.

---

## Один вход — много сервисов

Пользователь взаимодействует с системой через **единый интерфейс**.

Сейчас это:

- Telegram

В будущем:

- голос
- локальная панель управления
- возможно web-интерфейс

Все интерфейсы должны работать через **единый внутренний маршрутизатор событий**.

---

## Разделение ролей

Система делится на уровни:

```

Interfaces
↓
Router
↓
Services
↓
Execution
↓
Memory

```

---

## Человек подтверждает действия

Все операции, изменяющие память системы, должны проходить через:

```

plan → preview → confirm → execute

```

Этот механизм уже реализован и должен сохраняться.

# 4. Основные подсистемы

## Interface layer

Принимает входящие сообщения.

Примеры:

- Telegram команды
- текстовые сообщения
- голосовые сообщения

Интерфейс **не содержит бизнес-логики**.

---

## Event Router

Центральный модуль системы.

Задачи:

- определить тип входящего события
- выбрать нужный сервис
- передать событие в обработку

Router является **ядром системы**.

---

## Services

Сервисы реализуют конкретные действия системы.

Планируемые сервисы:

### Note service

Запись наблюдений в журнал.

---

### Observe service

Запись наблюдений по конкретному животному.

---

### Weather service

Работа с погодой:

- текущая погода
- предупреждения
- подписки

---

### Protocol service

Работа с ветеринарными протоколами.

---

### Limiter service (будущий модуль)

Учёт:

- кормов
- расходов
- ресурсов

---

## Execution layer

Исполняет действия.

Сейчас это:

```

file_executor.py

```

В будущем могут появиться:

- database executor
- sensor executor
- automation executor

---

## Memory layer

Слой хранения данных.

Включает:

### Farm memory (Markdown)

```

farm_memory/

```

Содержит:

- карточки животных
- журналы
- протоколы
- справочные данные

---

### Runtime storage

```

storage/

```

Содержит:

- pending.json
- weather_subscribers.json

---

### Event log (будущий модуль)

Будет хранить:

- входящие события
- распознанные намерения
- созданные планы
- выполненные действия

Это необходимо для:

- диагностики
- обучения системы
- анализа истории.

# 5. Этапы развития

## Этап 1 — стабилизация архитектуры

Цель:

подготовить систему к масштабированию.

Задачи:

- ввести единый объект **Event**
- добавить **router**
- перенаправить handlers через router
- не менять текущую бизнес-логику

---

## Этап 2 — разделение сервисов

Цель:

разгрузить controller.py.

План:

разделить логику на сервисы:

```

services/
note_service.py
observe_service.py
protocol_service.py

```

---

## Этап 3 — единый вход текста

Система должна понимать **свободный текст**.

Например:

```

У Маши сопли

```

вместо:

```

/observe Маша сопли

```

Router будет определять тип действия.

---

## Этап 4 — голосовой интерфейс

Добавление:

```

voice → speech-to-text → router

```

---

## Этап 5 — лимитер ресурсов

Добавление сервиса учёта:

- кормов
- закупок
- расходов

---

## Этап 6 — интеграция LLM

AnythingLLM будет использоваться для:

- анализа наблюдений
- рекомендаций
- помощи в диагностике

Но **не должен заменять бизнес-логику системы**.

---

# 6. Технические правила проекта

1. Не размещать бизнес-логику в handlers.
2. Все действия должны проходить через router.
3. Markdown остаётся основной памятью системы.
4. Все изменения памяти проходят через plan-confirm-execute.
5. Любой новый модуль должен быть сервисом.

---

# 7. Ближайшие задачи

Следующие шаги разработки:

1. Ввести внутренний объект `Event`.
2. Создать модуль `router.py`.
3. Перенаправить handlers через router.
4. Подготовить структуру папки `services/`.

Эти изменения не должны менять текущую функциональность системы.

---

# 8. Долгосрочное направление

Farm Guardian должен стать системой, которая:

- фиксирует события жизни фермы
- помогает принимать решения
- хранит знания хозяйства
- объединяет данные животных, погоды и ресурсов

Система должна оставаться **простой, локальной и надёжной**.
